## Короткий лид

Я собрал исследовательский стенд для сравнения Kubernetes HPA и внешнего MPC-контроллера: управляемый Go-сервис, Prometheus-метрики, Helm, HPA baseline, Python-контроллер, offline simulation, Grafana и воспроизводимые артефакты.

Итог оказался не рекламным “MPC победил HPA”. Гораздо интереснее: относительно консервативного CPU-HPA60 конкретный гибридный контур Hybrid-SA снизил среднее число реплик и ресурсную proxy-оценку, но в агрегированной серии ВКР короткий spike-сценарий проиграл по хвостовым задержкам. То есть результат — карта trade-off, а не универсальная победа predictive scaling.

В статье расскажу, как устроен стенд, зачем понадобилась QP/MPC-модель, как я сравнивал контроллеры и какие выводы из этого реально можно делать.

Репозиторий: <https://github.com/vshulcz/mpc-autoscaler>  
Демо-страница: <https://vshulcz.github.io/mpc-autoscaler/>

Важно: публичный репозиторий сейчас показывает узкий single-run snapshot для быстрой проверки пайплайна. Эта статья опирается на более широкую агрегированную серию из ВКР (`n=8`, `maxReplicas=70`). Поэтому числа в статье и текущий 60-second walkthrough отвечают на разные вопросы: walkthrough показывает репозиторный smoke-test, статья — итоговый экспериментальный trade-off.

## Почему вообще смотреть на MPC для autoscaling

Горизонтальное автоскалирование — это постоянный компромисс. Если держать много реплик, сервис спокойнее переживает пики, но ресурсы простаивают. Если держать мало реплик, средняя стоимость ниже, но при резком росте нагрузки появляются хвостовые задержки, таймауты и провалы успешности.

В Kubernetes стандартная опорная точка — Horizontal Pod Autoscaler. В типовой конфигурации HPA реагирует на уже измеренную метрику, например CPU. Это хорошо работает при плавных изменениях, но в коротких всплесках есть несколько источников запаздывания:

- метрику нужно собрать;
- HPA должен принять решение;
- Kubernetes должен запустить новые Pod;
- Pod должны стать Ready;
- сервис должен успеть принять дополнительную ёмкость.

Идея MPC здесь естественная: если есть короткий прогноз спроса, можно выбирать число реплик не только по текущей метрике, а по ожидаемой нагрузке на ближайшем горизонте. В моём случае это не “нейросеть для Kubernetes”, а более скучная и прозрачная QP-задача: штрафуем накопленную перегрузку, резкие изменения масштаба и лишнюю ёмкость. В статье я называю проверенный вариант Hybrid-SA; это конкретная инженерная конфигурация, а не утверждение обо всём классе MPC-методов.

Но важная оговорка: сама идея прогноза не гарантирует выигрыш. В Kubernetes полный контур включает телеметрию, планировщик, readiness, CPU limits, HPA behavior, Prometheus scrape interval и сетевые эффекты. Поэтому я строил не proof-of-concept на графиках, а стенд, где можно воспроизводимо проверить полный loop.

## Что построено

Репозиторий `mpc-autoscaler` состоит из нескольких частей:

| Компонент | Роль |
| --- | --- |
| `toy-load` | Go HTTP workload с настраиваемыми CPU, sleep, jitter, payload и error-rate. |
| Prometheus metrics | Request rate, latency histogram, in-flight requests, work knobs, error counters. |
| HPA baseline | Реактивное масштабирование через стандартный Kubernetes HPA. |
| MPC controller | Python-контроллер, который оценивает спрос, решает QP и рекомендует или применяет число реплик. |
| `analysis/` | Offline simulation, trace validation, summarizers, resource proxy summaries. |
| Grafana/dashboard | Визуализация request rate, latency, in-flight, CPU/memory и controller behavior. |
| Evidence docs | Индексы артефактов, benchmark matrix, limitations, rebuild commands. |

Системно это выглядит так:

```text
load profile -> toy-load -> Prometheus metrics -> HPA / MPC -> Kubernetes scale target
                                                    |
                                                    v
                                      run artifacts -> analysis -> summaries / figures
```

По умолчанию контроллер безопасен для экспериментов: offline-команды не трогают Kubernetes, а `mpc-control-loop` не масштабирует deployment без явного `--apply`.

## Как устроена MPC/QP-модель

Модель работает в дискретном времени с шагом управления. На каждом шаге есть текущее число реплик `r_t`, прогноз спроса на горизонте и внутреннее аппроксимирующее состояние перегрузки `b`.

В упрощённом виде цель такая:

```text
минимизировать: перегрузка + резкость изменения масштаба + стоимость ёмкости
```

В дипломной записи это выглядит как:

```text
J = sum_k (alpha * b_k^2 + beta * (x_k - x_{k-1})^2 + gamma * x_k)
```

Где:

- `b_k` — виртуальный накопитель дефицита целевой ёмкости;
- `x_k` — планируемое число реплик внутри горизонта;
- `alpha` — штраф за перегрузку;
- `beta` — штраф за резкость изменения числа реплик;
- `gamma` — штраф за лишнюю ёмкость.

Почему не моделировать p95/p99 напрямую? Потому что хвостовые задержки зависят от сервиса, очередей, CPU throttling, сетевого окружения и поведения Kubernetes. Поэтому в QP используется proxy-риск перегрузки, а реальные p95/p99 проверяются уже на полном стенде.

Важная граница модели: `b_k` не является настоящей очередью и не является оценкой p95. Это инженерная аппроксимация, которая помогает оптимизатору помнить о дефиците ёмкости. Все практические выводы должны проверяться экспериментально.

## Экспериментальная методика

Я сравнивал не изолированную QP-функцию, а полные контуры управления в Kubernetes.

Основные сценарии нагрузки:

| Сценарий | Профиль | Что проверяет |
| --- | --- | --- |
| Step | 20 RPS -> 80 RPS -> 40 RPS | Реакция на устойчивое изменение нагрузки. |
| Spike | 20 RPS -> 200 RPS на 30 секунд -> 20 RPS | Реакция на короткий всплеск. |
| Seasonality | синусоидальный профиль 20--120 RPS | Плавное изменение спроса. |

Финальная серия сравнивала CPU-HPA60 и Hybrid-SA при общем потолке `maxReplicas=70`. Для каждого сценария было по 8 повторов. Задержки в Kubernetes имеют тяжёлые хвосты и чувствительны к редким выбросам, поэтому результаты интерпретировались по медианам и IQR, без заявления формальной статистической значимости.

Метрики:

- request success ratio;
- p95, p99 и max latency;
- throughput по фазам;
- среднее число реплик;
- полная вариация масштаба `V`;
- proxy-оценка стоимости по requested CPU/memory.

Отдельно проверялись диагностические режимы:

- CPU-HPA target grid;
- Vanilla-HPA80;
- `proxy-HPA+safety` без QP-рекомендации;
- вариант “без QP”;
- Hybrid-Common с единым усреднённым профилем.

Это важно: если сравнить MPC только с плохо настроенным HPA, легко получить красивый, но бесполезный результат. Поэтому в работе есть не только HPA60, но и диагностические контуры.

- среднее число реплик ниже на 67.5%;

- `V` ниже с 110 до 97;
- успешность остаётся 100%;

Поэтому корректный вывод такой: Hybrid-SA даёт ресурсный компромисс относительно консервативного HPA60, но не доминирует по задержкам.

## Почему Spike оказался проблемным

Короткий spike — самый неприятный сценарий для predictive control в реальном Kubernetes-контуре.

В проверенной конфигурации:

- период управления Hybrid — 5 секунд;
- окно `rate()` для Prometheus — 15 секунд;
- scrape interval — 30 секунд;
- сама стрессовая фаза Spike длится 30 секунд.

Получается неблагоприятная наблюдаемость: несколько управляющих тактов могут опираться на один и тот же срез данных. Даже если контроллер решает “масштабироваться”, новые Pod не становятся готовыми мгновенно.

Проверка журналов показала один видимый системный фактор: в Spike медианное окно между применённой командой и готовыми Pod было около 40 секунд. Для 30-секундного пика это слишком долго. В одном повторе лаг перекрывал 20 секунд из 30-секундной стрессовой фазы, а p95/p99/max в этой фазе были 172.5/251.9/398.7 мс против около 22 мс в соседних фазах.

Это не доказывает единственную причину регрессии: лаг готовности Pod хорошо виден в журналах, но его нельзя полностью отделить от устаревания телеметрии, решений контроллера и возможного CPU throttling. Более аккуратный вывод: в полном контуре короткий всплеск может закончиться раньше, чем новая ёмкость реально появится, поэтому качество наблюдаемости и safety layer становятся не менее важны, чем QP-решение.

## Почему просто поднять target CPU у HPA тоже не спасает

Можно сказать: “HPA60 слишком консервативный, возьмём HPA150/HPA350 и сэкономим реплики”. Я это тоже проверял.

Диагностическая CPU-HPA-сетка с `maxReplicas=12` показала: рост `averageUtilization` действительно снижает среднее число реплик, но начиная с HPA150 в Spike появляются массовые таймауты. В точке HPA350 медианная успешность падает до 34.42%.

Это важный результат: простое повышение целевой CPU-утилизации может выглядеть ресурсно эффективно, но переводит контур ниже приемлемого качества. Поэтому сравнение autoscaling policy нельзя сводить только к “сколько реплик в среднем”. Нужны успешность, хвостовые задержки, replica dynamics и явные caveats.

```bash
python3 -m pip install -e analysis
mpc-validate-trace --trace-csv analysis/mpc_autoscaler_analysis/data/traces/baseline_spike_profile_dt15.csv

  --trace-csv analysis/mpc_autoscaler_analysis/data/traces/baseline_spike_profile_dt15.csv \
  --out-dir analysis/out/offline/spike
```

Этот путь не воспроизводит live Kubernetes результат сам по себе, но проверяет trace schema, simulator behavior и controller assumptions.

Полная структура воспроизводимости разделена на уровни:

| Уровень | Нужен Kubernetes | Зачем |

## Чего этот результат не доказывает

Это важная часть.

Работа не доказывает, что MPC вообще лучше HPA. Она не доказывает production-readiness. Она не даёт универсальных настроек для произвольных Kubernetes-кластеров.

Ограничения:

- workload синтетический;
- результаты зависят от Kubernetes-кластера, Prometheus scrape interval, readiness lag и CPU limits;
- HPA baseline чувствителен к target utilization и behavior settings;
- MPC зависит от качества demand estimate и параметров QP;
- public repo сейчас содержит узкий results snapshot, а не полноценный aggregate benchmark suite;
- сырой evidence archive не весь хранится в Git, чтобы не тащить тяжёлые артефакты.

Для меня это не недостаток публикации, а нормальная граница интерпретации. В autoscaling слишком легко получить красивую диаграмму, если не показывать, где именно контур сломался.

## Короткий вывод

Главный итог работы для меня не “MPC победил HPA”. Главный итог: конкретный Hybrid-SA-контур нужно оценивать как полный Kubernetes control loop, а не как красивую оптимизационную задачу отдельно от телеметрии, readiness lag и safety layer.

Hybrid-SA показал, что можно сильно снизить ресурсный резерв относительно консервативного HPA60 при 100% успешности, но короткий Spike выявил слабое место: новая ёмкость может прийти слишком поздно. Реактивные слои и качество наблюдаемости оказались не менее важны, чем QP-ядро.

Для меня это хороший результат: стенд стал не витриной “алгоритм лучше”, а инструментом, который показывает, где predictive scaling действительно полезен, а где ломается.
